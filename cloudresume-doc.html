AWS Cloud Resume Challenge

Building the Static Frontend

  • Website Development

    With no prior experience in HTML, CSS, JavaScript, or frontend development, I first dedicated time to learning the fundamentals. After gaining a basic understanding, and although I was not yet confident in building a website entirely from scratch, I explored portfolio templates on ThemeWagon and customized one to match my content and personal style.

  • Hosting the Static Website on Amazon S3

    This part was exciting because it marked the moment I finally began using the cloud in a practical way. To host my static website on AWS, I took the following steps:

    • Created an S3 bucket with a selected bucket name and AWS Region to store all website files.
    • Blocked all public access to ensure the bucket remained secure.
    • Instead of enabling S3 static website hosting, CloudFront was used as the content delivery layer to improve security, performance, and provide HTTPS support.

  • Setting up Content Delivery Network (CDN) using Amazon CloudFront

    I set up a CloudFront distribution to deliver my website content globally, ensuring it is served from edge locations with minimal latency. I configured the distribution's viewer protocol policy to HTTPS only for secure connections. Since my S3 bucket has all public access blocked, I used Origin Access Control (OAC) to restrict access so that only CloudFront can retrieve the content, following AWS best practices.

  • Setting the Domain Name System (DNS) using Hostinger

    After creating the CloudFront distribution, AWS provided a default domain name, but I opted to use a custom domain to create a professional and memorable URL. While AWS offers Route 53 as its DNS service, I chose Hostinger for managing alfredcm.cloud because it offered a more cost-effective solution. On Hostinger, I configured the DNS records, including CAA, CNAME, and ALIAS, to correctly point to my AWS resources.

  • Requesting an SSL/TLS Certificate in Amazon Certificate Manager (ACM)

    To add an alternate domain name to my CloudFront distribution, I first requested a public certificate from ACM in the us-east-1 region. I included a wildcard (*) in the fully qualified domain name so the certificate could cover future subdomains like blog.alfredcm.cloud or app.alfredcm.cloud. ACM provided a unique CNAME record for validation, which I added to my DNS provider (Hostinger). Once ACM detected the matching CNAME, it verified my domain ownership and issued the SSL/TLS certificate for CloudFront.

After working through all these steps in this stage, I was excited to see my first website officially hosted in the cloud.

Building the Serverless Backend

  • Adding a DynamoDB Table to Store the Visitor Count

    To store and update the number of visitors to my website, I provisioned a DynamoDB table to serve as the backend database for the counter. I configured it with a single partition key and added an initial item representing the visitor count, starting at zero.

  • Lambda Function for Visitor Counter

    Instead of accessing DynamoDB directly from the frontend, I created a Python Lambda function using the boto3 library. The function retrieves the current count, increments it, updates the database, and returns the new value to the frontend.

  • Security & Permissions (IAM)

    I configured a specific IAM Execution Role. Since Lambda functions are isolated by default, I attached a policy granting the function explicit permissions to read and write to my specific DynamoDB table, ensuring secure interactions.

  • Testing

    Before integrating with the frontend, I validated the function's logic using the AWS Console Test Event feature. I confirmed success by invoking the function and physically verifying in the DynamoDB console that the item count incremented from 0 to 1.

To tackle this stage, I had to take on the challenge of learning Python from scratch, as I had no prior experience with the language. I dedicated time to studying the basics, but with only foundational knowledge, I was not yet able to write the code independently. By leveraging online resources and going through a process of trial and error, I was eventually able to successfully develop the Python code needed for the Lambda function.

Frontend / Backend Integration

  • API Exposure (Lambda Function URL)

    To expose the backend logic to the public internet, I utilized the Lambda Function URL feature. I configured the Function URL to provide a dedicated HTTPS endpoint that is directly mapped to the Lambda function.

    To ensure security at the browser level, I enabled Cross-Origin Resource Sharing (CORS) in the Function URL settings. In the Allow Origins field, I specified my website's full domain, https://resume.alfredcm.cloud, so that only requests originating from my site are permitted.

    This setup ensures that while the endpoint is publicly accessible, it can only be successfully invoked by browsers running my specific website, effectively preventing unauthorized sites from inflating the visitor counter.

  • Integrating the Frontend with Backend

    I wrote a code wherein, when a visitor loads the website, the static JavaScript file executes an asynchronous HTTP request to the Lambda Function URL in order to retrieve and update the current visitor count. The response is then processed and dynamically displayed on the webpage, allowing real-time interaction between the frontend and the serverless backend.

Automation & CI/CD Processes

To ensure the project is maintainable and scalable, I implemented a full Continuous Integration and Continuous Deployment (CI/CD) pipeline. I used GitHub for version control and GitHub Actions to automate the deployment of both frontend website and the backend infrastructure. You can explore the full project source code in my GitHub repository here: https://github.com/alfredcmaghanoy/Cloud-Resume.git

  • Source Control Strategy (GitHub)

    The GitHub repository serves as the Single Source of Truth for the project. The HTML, CSS, and JavaScript files are version-controlled here. This allows me to track changes, revert to previous versions if something breaks, and work on new features without affecting the site immediately.

    I utilized a standard branching workflow. Code is developed and tested on a feature branch before being merged into the main branch.

  • The CI/CD Pipeline

    I configured a GitHub Actions workflow to automate the deployment of my static website. This ensures that every time I update my resume text or styling, the live site updates automatically.

    • The Workflow Process:
      1. Development - I make changes to the website code on my local directory.
      2. Push - I commit these changes and push them to the branch named update.
      3. Pull request - I review the changes made on the update branch before merging to the main branch.
      4. Trigger - The merge event triggers a workflow defined in .github/workflows/front-end-cicd.yml.
      5. Job Execution
        • Checkout - The GitHub runner checks out the latest code.
        • Configure AWS Credentials - The runner authenticates with AWS using secure GitHub Secrets (storing the Access Key ID and Secret Access Key securely, rather than in the code).
        • Sync to S3 - The workflow runs the AWS CLI command to upload only the modified files to the S3 bucket.
        • Invalidate Cache - The workflow clears the CloudFront cache, forcing the CDN to fetch the new files from S3 immediately so visitors see the latest updates.
  • Security Considerations

    I strictly avoided hardcoding the AWS credentials in the repository. Instead, I used GitHub Actions Secrets to inject credentials into the pipeline environment only during runtime.

Implementing Infrastructure as Code

I used Terraform to automate the backend infrastructure, ensuring consistent, repeatable, and version-controlled cloud deployments. Terraform was specifically applied to provision and manage AWS Lambda, DynamoDB, IAM roles and policies, and the Lambda Function URL for the page view counter API.

A dedicated provider.tf file was configured to define the AWS region and credentials, allowing Terraform to authenticate with my AWS account and align with the existing S3 bucket used for the frontend. While the frontend resources were deployed through a separate CI/CD pipeline, Terraform ensured that all backend services were reliably created, updated, and destroyed using code.

This approach reduces manual configuration, minimizes deployment errors, and enforces security best practices such as least-privilege IAM roles and restricted DynamoDB access.
